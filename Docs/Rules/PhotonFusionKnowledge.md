# Photon Fusion Knowledge Base (ナレッジベース)

このドキュメントは、Photon Fusion に関する仕様や、本プロジェクトを進める上で得られた知見、重要な情報、不要な情報などを蓄積・整理するための生きたドキュメント（Living Document）です。

## 1. 重要な情報 (Important Information)
プロジェクトのコアとなる、必ず守るべきルールやアーキテクチャの前提です。

* **サーバーによる物理演算の集中管理**:
  * 物理演算（衝突判定や角度計算など）はすべて Fusion のサーバー（Dedicated Server）が行っています。
  * WebGLクライアントはサーバーでの計算結果（Transform情報など）を受け取り、それを画面に描画（補間）しているだけです。
* **ネットワークトポロジー (Server/Client Mode)**:
  * WebGLはブラウザのセキュリティ（Socket通信の制約）によりHostになれないため、Server Mode（専用サーバーモード）を採用しています。
  * サーバーは画面を持たない Headless Server として動作し、全オブジェクトの状態権限（State Authority）を持ちます。
* **権限管理 (State / Input Authority)**:
  * **State Authority**: オブジェクトの状態を決定する絶対的な権利。本プロジェクトでは**常にサーバー**が保持します。
  * **Input Authority**: 各プレイヤーが自分のキャラクターに対して保持し、「前進する」などのキー入力（NetworkInput）のみをサーバーに送信します。

## 2. 必要な情報 (Necessary Information / Best Practices)
開発・実装時に参照すべきベストプラクティスや仕様の詳細です。

* **Tick ベースのシミュレーション**:
  * Fusion は固定時間のティック（例: 60Hz）でシミュレーションを進めます。Unity の `FixedUpdate` の代わりに、`FixedUpdateNetwork` メソッド内でサーバーが物理計算を更新します。
* **クライアントサイド・プレディクションと補間**:
  * プレイヤー自身のキャラクターは操作の遅延（ラグ）を隠蔽するため、クライアントサイド・プレディクション（予測）を用いて先行して動かします。
  * 動く机や椅子などはプレイヤーが直接操作しないため、予測は行わず「スナップショット補間（Snapshot Interpolation）」のみで滑らかに描画させます。
* **サーバー物理演算のカスタマイズ性**:
  * 専用サーバー（Dedicated Server）は「画面を描画しないUnityアプリケーション（Headless Build）」であるため、サーバー内の物理演算は**100%カスタマイズ可能**です。
  * `Project Settings > Physics` の変更（反発係数、摩擦、Solver Iterationsなど）はそのままサーバーに適用されます。
  * `FixedUpdateNetwork` 内で独自の重力処理や、不要なオブジェクトの物理演算のスリープ化、一部の小さな破片（紙切れなど）の計算をサーバーから除外してクライアント側でのみ「フェイク物理」として動かすといった最適化も自由に実装可能です。

## 3. 不要な情報・非推奨 (Unnecessary / Deprecated Information)
プロジェクトで採用しなかったアプローチや、誤りがちな設定などの備忘録です。

* **Host Mode / Shared Mode の使用**:
  * WebGL環境では Host Mode は使用不可。
  * 物理演算を伴う大量のオブジェクトの同期において、Shared Mode は各クライアントに権限が分散するため同期が破綻しやすく、本プロジェクトでは採用しません。
* **クライアント側での物理演算（Rigidbodyの直接操作）**:
  * クライアント側で独自に Rigidbody に力を加えたり、Transformを直接書き換えたりすることは同期ズレの大きな原因となるため禁止（非推奨）です。
